Offers¶

This package contains both offer-related data models: Offer and OfferTemplate

Offer¶

class cms.models.offers.offer.Offer(*args, **kwargs)[source]¶

Bases: django.db.models.base.Model

The offer model is used to activate an OfferTemplate for a specific Region. Integreat offers are extended features apart from pages and events and are usually offered by a third party. In most cases, the url is an external API endpoint which the frontend apps can query and render the results inside the Integreat app. The sole existence of an offer object instance means that it is activated, and if there is no object for a region/template-combination, it is deactivated.

Parameters:
  • id (django.db.models.AutoField) – The database id of the offer
  • created_date (django.db.models.DateTimeField) – The date and time when the offer was created
  • last_updated (django.db.models.DateTimeField) – The date and time when the offer was last updated

Relationship fields:

Parameters:
  • region (ForeignKey to Region) – The region where the offer is activated (related name: offers)
  • template (ForeignKey to OfferTemplate) – The template which is used for the offer (related name: offers)

Reverse relationships:

Parameters:feedback (Reverse ManyToOne Relation from OfferFeedback) – Feedback to this offer

property post_data[source]¶

In case the url expects additional post data, it is stored inside the post_data-dict. Some offers depend on the location which is realized by adding the postal code of the current region to the request. If the offer template indicates that the postal code should be used as GET-parameter, the class attribute use_postal_code has to be set to postal_code.POST (see cms.constants.postal_code) and then the key search-plz is automatically added to the post data. In case a third party service needs a different format, it has to be hard-coded here or we need other changes to the offer model.

Returns:The post data of the offer’s url
Return type:dict

property url[source]¶

The offer should inherit the slug property from its template. This is the url to an API endpoint in most cases. Some offers depend on the location which is realized by adding the postal code of the current region to the request. If the offer template indicates that the postal code should be used as GET-parameter, the class attribute use_postal_code has to be set to postal_code.GET (see cms.constants.postal_code) and the url has to end with the name of the required parameter-name, e.g. https://example.com/api?location=.

Returns:The url of an offer
Return type:str

Offer Template¶

class cms.models.offers.offer_template.OfferTemplate(*args, **kwargs)[source]¶

Bases: django.db.models.base.Model

The OfferTemplate model is used to store templates of offers which can be activated for specific regions. The information stored in an offer template is global, so if you need parameters, which depend on local information of a region, it has to be added to the Offer model.

Parameters:
  • id (django.db.models.AutoField) – The database id of the offer template
  • name (django.db.models.CharField) – The name of the offer template
  • slug (django.db.models.SlugField) – The slug of the offer template
  • thumbnail (django.db.models.URLField) – The thumbnail url of the offer template
  • url (django.db.models.URLField) – The url of the offer template. This will be an external api endoint in most cases.
  • post_data (django.contrib.postgres.fields.jsonb.JSONField) – If additional post data is required for retrieving the url, it has to be stored in this dict.
  • use_postal_code (django.db.models.CharField) – If and how the postal code should be injected in the url or post data (choices: cms.constants.postal_code)

Reverse relationships:

Parameters:
  • offers (Reverse ManyToOne Relation from Offer) – All offers which use this template
  • created_date (django.db.models.DateTimeField) – Created date
  • last_updated (django.db.models.DateTimeField) – Last updated
